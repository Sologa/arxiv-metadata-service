# Boolean Backend Spec

Status: initial backend implementation complete

## Purpose

Add a source-specific arXiv SQLite backend that can execute an already parsed
Boolean query object against the local arXiv metadata database.

This repo is responsible for layer 3 only:

```text
1. Parser layer
   Raw query string -> source-neutral AST.

2. Field mapping / capability layer
   Source-neutral field names and aliases for arXiv, S2, ACL, and other
   metadata sources.

3. Backend compiler / DB layer
   Parsed query object -> arXiv SQLite FTS and SQL predicates.
```

## Current State

The current database has these relevant tables:

- `papers`
- `paper_categories`
- `title_fts`
- `import_runs`

`papers` stores title, authors, abstract, categories, DOI, journal reference,
update date, and first version date. The current FTS index only covers title.

The current `/search` and CLI search behavior treats user input as plain title
text and rejects advanced FTS syntax such as column filters, prefix wildcards,
`NEAR(...)`, and grouped Boolean expressions. That behavior should remain stable
unless a separate advanced path is explicitly added.

## Scope

In scope:

- Add a backend API that accepts an already normalized query object.
- Add or rebuild an FTS5 table that can search fields beyond title, at minimum:
  - title
  - abstract
  - authors
- Compile text conditions to FTS5.
- Compile category and date conditions to normal SQL predicates.
- Preserve exact category matching through `paper_categories`.
- Preserve existing title search and candidate search behavior.
- Add tests that exercise the backend with parsed query objects, not raw query
  strings.

Out of scope:

- Raw query string parsing.
- Source-neutral parser grammar.
- S2 or ACL field mapping.
- Live arXiv API client behavior.
- Full compatibility with every arXiv API field.
- Changing `/Users/xjp/Desktop/Paper_DB/CLI_MANUAL.md`.

## Input Contract

The backend should accept a small normalized query object. The exact shape can
evolve, but the first version should be source-neutral enough for parser and
mapping layers to reuse.

Example:

```json
{
  "op": "AND",
  "children": [
    {
      "op": "OR",
      "children": [
        {"field": "title", "match": {"type": "phrase", "value": "survey"}},
        {"field": "abstract", "match": {"type": "term", "value": "taxonomy"}}
      ]
    },
    {"field": "category", "match": {"type": "term", "value": "cs.CL"}}
  ]
}
```

Supported operators for the first version:

- `AND`
- `OR`
- `NOT` or `ANDNOT`, if implementation can keep the semantics explicit

Supported field families for the first version:

- text fields: `title`, `abstract`, `authors`, `all`
- metadata fields: `category`, `primary_category`, `update_date`,
  `first_version_date`, `arxiv_id`

Supported text match types for the first version:

- `term`
- `phrase`

Range filters should be represented as structured objects rather than parsed
from strings in this repo:

```json
{
  "field": "update_date",
  "range": {"gte": "2024-01-01", "lte": "2026-12-31"}
}
```

## Backend Semantics

Text conditions should be compiled to FTS5 over a multi-column FTS table. The
backend may use column filters where the query can stay within one FTS
expression.

Category conditions should be compiled to SQL against `paper_categories`.

Date conditions should be compiled to SQL against `papers.update_date` or
`papers.first_version_date`.

Mixed Boolean expressions that combine FTS and metadata conditions should not
be forced into raw FTS syntax. The backend should build row-id sets with SQL
subqueries, CTEs, `UNION`, `INTERSECT`, or `EXCEPT` where needed.

## Non-Goals

The backend does not need to produce exactly the same ranking or result set as
the live arXiv API. It should provide a local, explainable, repeatable subset
that supports batch query testing without arXiv API rate limits.

Known differences from live arXiv API:

- The local snapshot may be older than live arXiv.
- Tokenization and phrase matching may differ.
- Ranking will use local SQLite FTS5 scoring or explicit local ordering.
- Some live API fields may be absent from the local snapshot schema.

## Initial Implementation Notes

The first implementation keeps the parser and source-neutral mapping out of this
repo. It adds:

- `paper_fts(title, abstract, authors)` while preserving `title_fts`.
- `ArxivSearch.boolean_search()` for normalized Boolean query objects.
- `POST /boolean-search` as a separate endpoint from `/search`.
- `arxiv-meta boolean-search` for normalized JSON query files.
- `arxiv-meta rebuild-paper-fts` to create or refresh `paper_fts` on an
  existing SQLite database.
- Dedicated generated fixtures under `tests/fixtures/boolean_backend/`.
- Dedicated backend tests under `tests/boolean_backend/`.

Version 1 supports `AND`, `OR`, and explicit `ANDNOT`. Negative-only `NOT`
remains out of scope until the parser/mapping layer specifies its semantics.
